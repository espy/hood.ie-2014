Hello there, Hoodies!

Last Wednesday to Friday, Max, Jens and Alex met in Berlin to work on the new hood.ie that's to be relaunched with Hoodie's 1.0 beta. Our aims for these three days were to:

  • find a new visual language and document it in a styleguide
  • structure all things Hoodie on the web in a useful way
  • make plans and draw up concepts for an on-boarding flow

We didn't quite make it through, but we got a lot done.

1Wednesday

Wednesday was spent going through the relaunch google doc, evaluating, weighting and rearranging all the information we'll be offering, planning out URL structures and how the first encounter with Hoodie on hood.ie should go for different sorts of people.

Infrastructure/Hosting

We decided to stay with gh-pages as an infrastructure, since its benefits vastly outweigh anything any CMS could offer us:

  • people can fork and PR content corrections, which has proven valuable in the past, and a CMS would really get in the way of that
  • we can run it on GitHub and don't have to worry about Hackernews spikes
  • deployment is simple and foolproof now that we've discovered grunt-gh-pages
  • it's less work

Note: We can get search via Google's Custom Search Engine or DuckDuckGo's Searchbox, which isn't as good though, because it's an external page and you can't style or embed it.

Hoodie/Start Page

We decided that hood.ie should function as a hub page, with many content tiles that give some indication of what lies behind them. The main focus of our energies should be the first four, which are:

  1. a header block, with title, subline, search box and video button for a 3-minute overview
  2. a feature/testimonial block, with a bit more info about what Hoodie can do for you, and 2–3 people telling us what Hoodie did for them
  3. a code showoff block that could use the autotyping thing I built ages ago. How to Hoodie:
    • Signing up a user
    • Storing private data
    • Retreiving data
  4. the try block. This is a bit more involved, but think of it as a flow chart:

The try-block

It starts with "Want to try Hoodie now?" and leads to three options:

  1. Yes, but I'm in a hurry! (No installation, super quick!)
  2. Yes, let me install it now! (You're comfy with the terminal, right?)
  3. Hmmm, I'd like to know more first… (Ok, let us convince you)

1. leads to try.hood.ie, which in turn leads to docs.hood.ie/tutorials/first-steps.

2. leads to the installation instructions.

3. leads to hood.ie/technology, where we explain the tech and how everything works, and then back to a choice of 1, 2 or docs.hood.ie.

The meat is in part 2: its the installation instructions for all platforms, followed by two buttons:

  1. Yay, everything works!
  2. Meh, something went wrong

1. leads to docs.hood.ie/tutorials/first-steps

2. leads to hood.ie/get-help

Note: This indicates a central idea behind the site: everything leads somewhere contextually sensible. No dead ends.

Everything else (contribute, get-help, about, technology, events-and-media, merch, contact, plugins, code-of-conduct, newsletter signup etc.) is teased via blocks of varying size, all of which should offer a glimpse of the actual content behind them.

  • You have money and vision? Sponsor a week of development!

The right 1/3 is reserved for the current sponsor (or a sponsoring CTA), and could display a fairly big sponsorship logo.

Special Blocks/Events and Media

Events and Media should probably be handled via Jekyll collections, we'd need to try that out. Every event and media exposure (podcast, interview, article etc.) would be a yml-file with some metadata (date, type, place, latlon, starred, videoLink, slidesLink etc.) and a markdown description. This would allow us so build a sweet, sortable and filterable events page that would automatically know which events were in the past and future, and which one is next. It would also allow us to build an events and media block on hood.ie that shows the next event and the last podcast, as well as the total number of events, podcasts, interviews and articles.

The rest would be very simple: title, description, maybe an image, linking to a simple html page with anchors, or away to another subdomain.

2Thursday

Thursday was spent on design: fonts, colours, layout options, header ideas and graphical flourishes.

We're all still big fans of Bariol regular and bold as our main fonts and Courier Prime as our monospace, Jens also had another excellent choice for copy text, but loading 4 font files may turn out to be a bit much.

As for the colours, we're trying to get away from using the current logo colours together, since they do tend to clash a lot. Jens' idea was to use single-coloured versions of the logo (all orange, for example), and Max added that we could use this approach to colour-code our most important site sections.

Apart from that, we're going for warm and welcoming tones, so no white backgrounds and blue text any more. As for colour use: we want more of it, and we've got a bunch of pastel shades that work well with our choice of background and the single main colour from the logo.

We came up with some playful signature elements, such as

  • pastel background dots and dashes that lie behind text, emphasising it or giving additional info (i.e. red dashes for warnings)
  • large individual elements of the logo appearing from beyond the edge of the page
  • a streamlined version of the logo, with less inner lines

All in all, we feel like we've got a very good direction, and we're happy with our basic choices and ideas, but we're still working on pulling it all together. Here's what we've got.

1Colors and their use

The original Hoodie colours have had a good run, but they've become increasingly problematic, since it's quite hard to base designs on them. One solution is to only use one of the colours as the signal colour for the current page, the other is to go with a new, desaturated colour palette. This will let us use more colours simultaneously without being as clashy and glaring as the old coulours would have been.

Brand Colors/Logo

Colors derived from the logo.

Note: These are rarely used to style other elements, since this approach has proved quite limiting. You'll notice that, when used together, they frequently clash with everything else, and even with each other.
